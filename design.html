1. Introduction

It is assumed that all users of the application have access to a modern browser with JavaScript and React support (virtually all popular mainstream browsers including Chrome, Safari, Firefox, etc.). The application is a progressive web application, accessible through modern web browsers.

1.1 Purpose

The purpose of the design document is to outline the architecture and system design of Gomoku and it's components

2. Planned Implementation

2.1 Choice of Language

Backend: The backend language of choice will be JavaScript (NodeJS) with the Express backend framework. NodeJS and Express provides an excellent environment for developing web applications, allowing for developers to quickly render application templates to HTML, as well as providing necessary facilities for interfacing with any needed dependencies.

Frontend: The frontend language of choice will be Javascript with the React frontend library. React is easy to use and very flexible, which is perfect for creating visually appealing, scalable and fast user interfaces. Additionally, the use of JavaScript in both frontend and backend ensures that developers are familiar with both frontend and backend code as well as seamlessly switch between the two.

2.1.1 Previous Planned Language

Backend: The previously planned backend language was Python with Flask. We decided to switch to NodeJS with Express due to it's versatility, speed, and to improve compatability with the frontend.

Frontend: The previously planned frontend language was C compiled with WASM. This was switched to JavaScript with the React library as it's easier to use and has improved compatibility with the backend.

2.2 Intermediate Server

After the initial MVP release an intermediate server will be required to implement multiplayer. The intermediate server will be implemented using JavaScript with SocketIO to facilitate bi-directional communication.

5. Detailed Design

5.1. Module Detailed Design

This section will describe the module design detail. Each interface and internal function which is implemented by a module or process will be described along with the steps followed to get from input to output, with the use of high level pseudocode. This pseudocode will be helpful in order to translate it into actual code during the implementation stage.

5.2. Grill class

In the Grill class, it most describes the attributes of the board of the Gomoku such as height, width, color and the point that has been chosen by an AI or human player. And it will detect if any player is winning or not.

5.3. AI class

The AI class contains the AI Gomoku algorithms to evaluate each step the AI takes,
and to decide which point to choose will generate the best result.

negamax(is_ai, depth, alpha, beta)
. Parameter:
  • is_ai: to flag if its AI's move
  • Depth: to define the depth of searching
  • Alpha: the max value that a pattern could have
  • Beta: the min value that a pattern could have
. To search which position to move will generate the best result
has_neighbor(node)
. Parameter
  • Node: the location of the chosen point
. To find if other node is adjacent to the given node
cal_score()
. To evaluate the move by giving each different situation a value,
and compare the current situation with those hardcoded situations.
evaluation()
. To calculate the difference between two player's score

6. Maintenance

  1. Adaptive: Gomoku is web-based application. Therefore it has to work perfectly on all mainstream browsers such as Safari, Chrome, Firefox, Internet Explorer, and Microsoft Edge, etc.
  2. Corrective: A “Report to Us” icon will be placed in our application interface that allows user to report any issues they encounter in the game. If it’s users’ personal issues, the implementation team will contact the user through email to provide feedback within 3 days. If there is a large amount of similar feedbacks, our implementation team will conduct temporary maintenance and upgrade to the whole website.
  3. Perfective: Due to time constraints, in order to ensure that the application can be successfully executed, we will give priority to the development of the main features and functionalities of the application. Once the program becomes more reliable through constant maintenance, we will develop and launch more features and functionalities to make the game more playable. In addition, a survey will be conducted on a regular basis to collect feedbacks to improve user experience. We will consider all things suggested by users and launch new features based on popularity of the recommendations.

Some examples of features that is on the shortlist to be implemented in the application is listed below:

- More colours and theme that allows user to customize by themselves

- Player’s historical record

- Auditorium: A feature that allows other users to watch the game
